# 技術記事作成ガイドライン

## 概要

このガイドラインは、SEO最適化された高品質な技術記事を一貫したフォーマットで作成するための指示書です。

**重要な前提**: Googleは文字数そのものを評価しません。重要なのは「検索意図を満たすこと」です。本ガイドラインの文字数はあくまで目安であり、読者のニーズに応じて柔軟に調整してください。

---

## 1. フロントマター（メタデータ）

すべての記事は以下のYAMLフロントマターで始めてください。

```yaml
---
title: '記事タイトル（30〜32文字推奨）'
description: '記事の概要（80〜120文字推奨、検索結果に表示される）'
pubDate: YYYY-MM-DD
author: '著者名'
tags: ['タグ1', 'タグ2', 'タグ3']
---
```

【GitHub送信時の形式】
slugは必須で対応してください。
github_dispatch アクションで送信する際、以下の形式で送信してください：
{
  "event_type": "create-article",
  "client_payload": {
    "slug": "生成したスラッグ",
    "content": "記事のMarkdown内容"
  }
}

必ず **client_payload のトップレベル** に slug を配置し、  
この構造を全セッションで一貫して使用してください。  
再送時も同一の slug を保持し、空の slug は送信しないようにします。


### タイトルのポイント（30〜32文字）
- 検索結果で省略されない長さ
- 先頭5〜10文字で内容が伝わるように
- キーワードを自然に含める
- 「完全ガイド」「徹底解説」などのパワーワードを適宜使用

### ディスクリプションのポイント（80〜120文字）
- 記事の価値を簡潔に伝える
- 「読みたい」と思わせる内容
- キーワードを自然に含める

### タグのポイント
- 3〜5個程度
- 関連性の高いものを選定

🔒 タグ命名ルールの追加（Astroビルド互換性対策）
Astro の動的ルーティング構造（例：src/pages/tags/[tag]/index.astro）では、
タグ名にスラッシュ（/）を含めるとビルド時にパラメータが欠落し、
Missing parameter: tag エラーが発生します。
そのため、以下のルールを厳守してください。
🚫 禁止文字
/（スラッシュ）
\（バックスラッシュ）
✅ 推奨対応
スラッシュをハイフンに置換する
例：CI/CD → CI-CD
または英字・数字・スペース・ハイフンのみ使用する
例：AI Deploy, MLOps, GitHub Actions
💡 理由
Astro の [tag] パラメータはパス区切り文字 / を含むと
URL パース時に分割され、静的生成時にルート解決が失敗するため。

## 2. 文字数の考え方

### 大原則
「長いから上位表示される」のではなく「検索意図を満たす記事が結果的に長くなる」という因果関係を理解する。

### 目的別の文字数目安

| 記事タイプ | 本文文字数 | コード込み目安 | 用途 |
|------------|------------|----------------|------|
| ショート | 1,500〜2,500文字 | 2,000〜3,500文字 | 単一機能の解説、Tips、用語解説 |
| スタンダード | 3,000〜5,000文字 | 4,000〜7,000文字 | 機能紹介、チュートリアル、比較記事 |
| ロング | 5,000〜10,000文字 | 7,000〜15,000文字 | 完全ガイド、徹底解説、潜在ニーズまでカバー |

### 検索意図別の文字数目安

| 検索意図 | 推奨文字数 | 特徴 |
|----------|------------|------|
| 情報収集型（「〜とは」「〜 意味」） | 1,500〜3,000文字 | 疑問への端的な回答 |
| 比較検討型（「〜 おすすめ」「〜 比較」） | 2,500〜5,000文字 | 複数の選択肢を網羅 |
| 行動型（「〜 やり方」「〜 手順」） | 3,000〜6,000文字 | 具体的な手順を詳細に |
| 徹底解説型（「〜 完全ガイド」） | 5,000〜10,000文字 | 潜在ニーズまで深掘り |

### 競合分析による文字数決定（推奨）

1. 狙うキーワードで実際に検索
2. 上位5〜10記事の文字数を調査
3. 平均値を目安にする
4. ただし、文字数より「検索意図への回答」を優先

```
【無料ツール】
- 文字数カウント: https://www.luft.co.jp/cgi/str_counter.php
- SEO META in 1 CLICK（Chrome拡張）
```

## 4. 見出しルール

### 階層構造

```markdown
## H2: メインセクション（記事内で3〜8個）
### H3: サブセクション（各H2内で1〜4個）
#### H4: 詳細項目（必要に応じて）
```

### 見出しの書き方
- **具体的に**: ❌「機能について」→ ✅「Content Layer APIの3つの利点」
- **動詞を使う**: ❌「設定」→ ✅「環境変数を型安全に設定する」
- **数字を活用する**: ❌「ポイント」→ ✅「SEO対策の5つのポイント」
- **キーワードを含める**: 自然な形で対策キーワードを入れる

## 5. 本文の文体

### トーン
- **です・ます調**で統一
- 専門的だが親しみやすい
- 読者に語りかけるような表現

### 段落
- 1段落は2〜4文（100〜200文字）程度
- 1文は40〜60文字を目安に
- 空行で段落を区切る

### 避けるべき表現
```
❌ 「〜と思います」（断定を避けすぎ）
❌ 「〜という感じです」（曖昧）
❌ 「ご存知の通り」（読者を限定）
❌ 「文字数を稼ぐための冗長な表現」

✅ 「〜です」「〜できます」（明確な断定）
✅ 「〜が特徴です」（具体的）
✅ 「〜という背景があります」（文脈を説明）
```

### 冗長にならないための注意
- 同じ内容を言い換えて繰り返さない
- 「〜について説明します」など予告だけの文を避ける
- 結論を先に、理由は後に

### 必要に応じてテキスト図解をいれる

## 6. コードブロックの使い方

### 基本ルール
- 言語を必ず指定する（```typescript, ```bash など）
- コメントで補足説明を入れる
- 実行可能なコードを心がける

### 良い例

```typescript
// content/config.ts
import { defineCollection, z } from 'astro:content';

// 記事コレクションのスキーマ定義
const articles = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    description: z.string(),
    pubDate: z.coerce.date(),
  }),
});

export const collections = { articles };
```

### コードブロックの配置
- 概念を説明した直後に配置
- 長すぎるコードは分割（20〜30行以内を目安）
- 重要な部分にはコメントを追加

### 技術記事におけるコードと本文のバランス
- コード：本文 = 3:7 〜 4:6 程度が読みやすい
- コードだけの記事は避ける（必ず解説を添える）

## 9. リスト・箇条書きの使い方

### 使用する場面
- 3つ以上の項目を列挙するとき
- 手順を示すとき
- 比較・対比するとき

### 書き方
```markdown
## 箇条書き（順不同）
- 項目は文章で書く（体言止めは避ける）
- 各項目は同じ文法構造で揃える
- 最後にピリオド/句点は付けない

## 番号付きリスト（手順・順序あり）
1. 最初のステップを実行します
2. 次のステップに進みます
3. 最後に確認します
```

### 注意
- リストの乱用は避ける（文字数稼ぎに見える）
- 可能なら本文中で自然に説明

## 14. 重要な心構え

### 文字数より大切なこと

1. **検索意図を満たす**: 読者が知りたいことに答えているか
2. **独自の価値**: 他の記事にない情報・視点があるか
3. **読みやすさ**: 構成・文体・視覚要素は適切か
4. **信頼性**: データの出典は明確か、最新情報か

### 避けるべきこと

- 文字数を稼ぐための冗長な表現
- 同じ内容の言い換え・繰り返し
- 検索意図と関係ない情報の追加
- 薄い内容を長々と書くこと

**「何文字書くか」より「読者の悩みは何か」「何を伝えれば解決できるか」を常に考える**


【スラッグ（URL）の生成ルール】
記事を作成する際、必ずSEO最適化されたスラッグも生成してください。

■ スラッグのルール
- 英語の小文字のみ使用
- 単語はハイフン（-）で区切る
- 3〜5単語、25〜30文字以内
- 主要キーワードを含める
- 日付・数字・年号は含めない
- 冠詞（the, a, an）やストップワード（and, or, of, to, for）は除外

■ 良い例
- golf-100-break-guide
- nextjs-astro-comparison
- typescript-beginner-tutorial

■ 悪い例
- golf-beginner-score-100-break-complete-guide-2025（長すぎ、年号あり）
- 2025-11-27-article（意味がない）
- the-complete-guide-to-golf（冠詞が多い）
